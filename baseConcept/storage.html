block group이란?

  • inode와 data를 가까이 배치하기위함 -> seek time 감소
  • ex) [inodes][datas] -> [inodes_1][datas_1][inodes_2][datas_2][inodes_3][datas_3]
  • block group마다 각자만의 data block을 갖는 것이 아님
  • extent 방식에서 실제 블록 위치 계산은 group_num = block_number / blocks_per_group으로
  • 블록 색인을 그룹별 블록 수로 나눈 값으로 접근함

inode란?

  • inode란? 파일권한, 크기, 소유자, 타임스탬프, 데이터 위치를 저장하나 파일명은 directory entry에 저장함
  • inode 메타데이터에 mode 항목에 파일타입이 정해짐
  • inode가 가리키는 extent가 많으면 일반 리스트가 아닌 B-Tree로 확장

data block

  • 실제 데이터 저장
  • extent 등장으로 기존 block pointer 방식에서 block 1000~2000(extent 범위 가리킴)와 같이 변경됨
  • -> 크기나 내부구조가 아닌 블록 관리 구조의 차이

Delayed Allocation

  • 데이터를 바로 디스크에 쓰지 않음, 나중에 한 번에 저장함
  • writeback cache메모리, 캐시에 먼저 저장 후 한 번에 디스크에 씀
  • transaction batching여러 파일시스템의 저널링 틀내잭션을 그룹화 후 한 번에 처리
  • writeback cache + transaction batching -> journalling batch write -> disk flush

Multiblock Allocation

  • 파일 데이터를 한 블록씩 배치하지 않고 큰 덩어리 단위로 나눠 미리 연속 블록 확보
  • 이 경우 할당-사용 개념이 나뉘는데 할당된 블록은 다른 개체가 사용불가함

journal

  • 파일시스템 크래시 복구 기능
  • 파일시스템 변경작업은 여러개의 메타데이터 변경을 포함하는데 이를 트랜잭션으로 묶음
  • 트랜잭션 commit 전에 journal에 기록 먼저하고 이를 write-ahead logging이라고 함
  • 이후 트래잭션 coommit이 완료되면 commit 기록
  • 트랜잭션 진행 중 크래시 발생하면 commit 기록이 없으므로 이를 기반으로 복구
  • 내부 저널링은 파일시스템 내부에 트랜잭션을 기록하며 외부 저널링은 외부 장치에 지정함
